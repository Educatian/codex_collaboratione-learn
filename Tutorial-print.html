01

What this tutorial is for

Scope, audience, prerequisites, and a suggested teaching sequence.

Use this guide when you want to work with Codex to create learning content that is interactive, instructional, testable, revisable, and deployable — either inside an LMS or on the open web.

This tutorial focuses on the whole process, not only the code:

  • defining the learning problem
  • specifying the interaction model
  • writing good prompts
  • reviewing intermediate results
  • refining the design and behavior
  • packaging for SCORM when needed
  • creating a separate web deployment version when needed
  • testing inside Moodle or another LMS
This is not only a coding guide. It is a collaboration guide.

Primary audience and assumed skill level

Audience priority

Primary: practicing instructional designers, learning designers, learning engineers, and technically curious educators who want to collaborate with Codex to build working digital learning objects. The document is operational first — it is written to be used, not only to be read. Secondary: graduate and upper-undergraduate instructional-design students using this document as a lab manual in a course. For that audience, the prompt schema (§5), the workflow (§6), and the delivery-target decision framework (§4) are the entry points; the appendices provide scaffolding for guided practice.

This tutorial is written primarily for practicing instructional designers, learning designers, learning engineers, and technically curious educators who want to collaborate with Codex to build working digital learning objects. A tertiary audience is front-end developers or educational software collaborators who need a clearer view of the instructional logic behind the build requests.

That means the document is not written only for professional developers, but it is also not written as a zero-technical-literacy guide. It assumes the reader can work with files, understand the difference between a web version and a SCORM package, and review outputs in a browser or LMS. Where the workflow becomes more technical — such as SCORM packaging or runtime state handling — the document tries to explain the reason for each step rather than only naming it.

If the reader is a classroom instructor or subject-matter expert with little implementation experience, the document can still be useful, but that reader may need developer support for the packaging and testing stages. If the reader is an experienced developer, the instructional-design framing may be the more valuable part. If the reader is a student, the teaching sequence further down this section is the recommended on-ramp.

Intended use context

This document was written with two concrete contexts in mind, and its examples, defaults, and tone reflect both:

  • Professional practice — higher-education learning-design teams, centers for teaching and learning, research groups building learning artifacts for courses or studies, and individual instructional designers who need a reproducible Codex workflow.
  • A graduate or upper-undergraduate course in a US instructional-design, learning-sciences, or educational-technology program — specifically, a course that covers interactive courseware production, LMS delivery, and iterative design-build cycles. The document is structured so that it can be decomposed into a five-module lab sequence (see the teaching sequence below) running roughly six to eight weeks of a standard semester.

Two things follow from this scope. First, institutional defaults reflect US learning-design realities across two primary contexts: US higher-education institutions (public and private) and US corporate learning & development (L&D). WCAG 2.1 AA (§22) is treated as a legal floor for both; the §16 LTI integration path is framed as the default for Canvas-first higher-ed programs; Cornerstone, Workday Learning, and Docebo are the reference points for corporate deployment (§15). Second, outside this scope — K–12 compliance training, non-US regulatory regimes, and vendor authoring platforms (Articulate, Rise, iSpring) — much of the workflow still transfers, but terminology and defaults may need adjustment.

02

Treat Codex like a build partner

The mindset that makes the workflow work.

The most useful mindset is this:

Anti-pattern

Codex should not be treated like a one-shot generator that magically outputs finished courseware from a single prompt.

Instead, treat Codex like a technical build partner who helps you through an iterative production process. That means:

  1. You define the learning goal clearly.
  2. Codex builds a first working version.
  3. You review it in context.
  4. You identify concrete issues.
  5. Codex refines the artifact.
  6. You test it in the target delivery environment.
  7. You preserve stable versions and split deliverables when necessary.

This process is much more reliable than trying to get a perfect final artifact in one shot.

2.1 AI authoring ethics and disclosure

Treating Codex as a build partner only works if the partnership is visible. An AI-assisted learning object is not a neutral artifact: learners, instructors, and institutions have legitimate questions about who wrote what, what was verified, and what data the tool saw. Address those questions before shipping, not after.

Minimum disclosure
  • State in the learner-facing UI or the course syllabus that the object was built with AI assistance, and note the human role (design, review, packaging).
  • Keep a short record of which prompts shaped which components, so later maintainers can see the provenance of the interaction logic.
  • Do not paste identifiable learner data, proprietary institutional material, or unreleased assessment items into prompts. Treat Codex conversations like any other third-party service that may log input.
  • Follow your institution's policy on AI-generated course materials. If there is no policy yet, write one for your own team and circulate it upward.

Academic integrity also runs in the other direction. If the object is used for assessment, the rubric should distinguish between "use AI as a study partner" and "submit AI output as your own analysis." This is a design question, not a disclaimer question — build the activity so that the honest path is the easy path.

2.2 FERPA, institutional AI policy, and compliance disclosure

US higher-education institutions operate under FERPA (20 U.S.C. §1232g), and US employers operate under sector-specific training-record requirements (OSHA 29 CFR 1910 for safety, HIPAA for healthcare, SOX for financial controls). An AI-assisted authoring workflow intersects both regimes in three predictable places:

  • Input data. Never paste personally identifiable student records (names, IDs, grades, disability accommodations) or protected HR records into a Codex prompt. Under FERPA, "school official" status does not extend to third-party AI services unless there is a written agreement articulating the legitimate educational interest and a functional control over the vendor. Treat Codex as an external vendor by default.
  • Generated assessment items. If Codex authored items that later contribute to a grade, document that fact. For higher-ed accreditors (SACSCOC, MSCHE, WSCUC, HLC, NWCCU) and corporate audit regimes, the evidence trail of "who authored what" is part of what makes an assessment defensible.
  • Learner-facing disclosure. A growing number of institutions require a syllabus statement or course-page notice when AI tools substantively contributed to course materials. Some faculty senates now distinguish "AI used to generate graphics," "AI used to draft text," and "AI used to generate assessment items" — disclose at the most specific level your policy reaches.
Disclosure template (higher-ed)

"Portions of this learning object were drafted with AI assistance (OpenAI Codex) under the direction of [designer]. All learning objectives, assessment items, and rubrics were reviewed and approved by a human subject-matter expert before release. No identifiable student information was shared with the tool."

Disclosure template (corporate L&D)

"This training module was developed with AI-assisted authoring. Content was reviewed for accuracy by [SME], for compliance alignment by [compliance lead], and for accessibility by [accessibility reviewer]. Authorship records and the prompt history are retained in [system] for the retention period required by [policy / regulation]."

If your institution has no written AI-in-courseware policy yet, write a one-page internal policy for your own team: what data can and cannot go into prompts, what must be disclosed to learners, who reviews AI-generated assessments before release, and where the prompt history is archived. Circulate upward. Absence of policy is the riskiest state — not because the tool is unsafe, but because "no one told me" is a weak defense in a grievance, audit, or Title IX / Title VI review.

2.3 Academic integrity toolchain

If the learning object is used in a course where the assessment counts, the integrity question runs both directions: learners should not submit AI-generated work as their own analysis, and instructors should not unknowingly embed AI-hallucinated content as the "correct answer." The tooling side of this conversation is worth a short inventory.

  • Turnitin. Historical strength is text-similarity detection; the current AI-writing indicator is a separate score and should be read as a prompt-for-review, not a verdict. Most US institutions with a campus-wide Turnitin license surface both scores to instructors; the honor-code hearing panel decides.
  • Copyleaks. Competing similarity + AI-detection platform with LMS integrations for Canvas, Blackboard, Moodle, Brightspace, and Schoology. Often adopted alongside or instead of Turnitin when institutions want broader multilingual coverage.
  • GPTZero and Originality.AI. Standalone AI-detection services commonly used by individual faculty. Their confidence scores are probabilistic — do not build a disciplinary case on a single detector's output.

Two design consequences for AI-assisted authoring:

  1. Do not publish AI-generated text as a reference answer that learners will then be asked to reproduce. That turns your tool into a closed loop: the learner copies, the detector flags, and the origin is obscured. Rewrite AI drafts through a human voice before shipping.
  2. Distinguish "practice with AI" from "submit without AI" at the activity level. A formative practice quiz where learners can use AI freely is a different object than a summative rubric-graded task. State the rule on the activity itself — not only in the syllabus — so the honest path is the easy one.

For provost-office and faculty-senate guidance on AI use in coursework, point colleagues to their own institutional policy first; in the absence of one, the MLA/CCCC Joint Task Force on Writing and AI and the AAC&U AI-in-undergraduate-education resources are common US reference points.

Design for return visits, not single sittings

For families B (practice), C (branching), and F (mini-game) in particular, durable learning depends on spaced retrieval — learners benefit more from returning to the activity across days than from one long session. This is a design request, not a magic property of the tool. When you prompt Codex, specify: state that survives a closed tab; entry points that let the learner resume mid-practice; and a visible cue that some items are due for review again. A quiz object that treats every session as a fresh start is quietly discarding the effect most responsible for retention.

Sections 9, 13, and 14 argue repeatedly that SCORM should not be the reflex default. This short table turns that argument into a decision rule you can apply before writing any prompt.

A practical note on local Moodle as a sandbox

One of the most useful practical lessons is that a local Moodle instance is often the right first sandbox for LMS testing. Not because Moodle represents every LMS perfectly — it does not — but because it is relatively accessible, testable, and repeatable. When you are trying to understand whether a coded learning object actually behaves like courseware, what you need first is a stable place to upload, launch, break, revise, and re-test. A local Moodle sandbox gives you that loop.

This is especially valuable when moving from prototypes into runtime questions such as: Does the package upload cleanly? Does it launch correctly? Does state persist? Does re-entry work? Do score and completion behave as expected? Those are implementation questions, and they are much easier to answer when you control the environment.

At the same time, Moodle should not be treated as a universal proxy for every LMS. A package that behaves well in Moodle may still behave differently in Canvas, Blackboard, or another platform. The right mindset is:

  • use Moodle as your first practical sandbox
  • use it to discover and fix obvious packaging and runtime issues
  • then validate more broadly if cross-platform compatibility matters

Articulate and code-authored content

A major theme in the conversations that produced this document was the question of whether code-authored learning content can substitute for tools like Articulate. That framing is worth including because it changes what you test for.

If you compare code-authored content to a traditional authoring tool, the real question is usually not "Can I make something visually similar?" It is "Can I reproduce enough of the invisible runtime work that established authoring tools already handle?" Tools like Articulate are not only slide builders. They also act as translators between content, packaging, state, reporting, and LMS runtime expectations.

A serious replacement workflow should test at least three layers: the interaction itself, the packaging and runtime behavior, and the maintainability of the content as it changes.

Maintainability as a test dimension

The third layer should not remain abstract. If maintainability is part of the comparison, it needs to be tested deliberately rather than mentioned rhetorically. One practical test is to build the same small learning object in two forms: a traditional authoring-tool version and a code-authored version. Then apply the same change requests to both.

Useful change requests include:

  • revise introductory copy
  • add one new quiz item
  • change the scoring rule
  • reorder one screen or state
  • localize key interface text
  • adjust the visual theme
  • alter one branching path

For each change, record time to implement, ease of review, visibility of what changed, rollback difficulty, mergeability with concurrent work, and regression risk. This is where code-authored content often shows a real advantage: source files can be versioned in Git, diffed line by line, reviewed in pull requests, and branched safely. A .story file, by contrast, is a binary authoring file — it can be backed up, but it is much harder to inspect or merge transparently. That does not automatically make code better in every context, but it makes maintainability testable rather than merely aspirational.

A more balanced comparison

The comparison should stay intellectually honest. There are several domains where Articulate remains genuinely strong.

Authoring tools often win when
  • slide-based narrative authoring
  • rapid assembly by non-developers
  • template-driven course construction
  • review workflows for non-technical stakeholders
  • fast prototyping of conventional compliance content
  • teams already standardized on a suite
Code-authored workflows often win when
  • unusual or highly custom interactions
  • visualization- or simulation-heavy work
  • fine-grained version control
  • deeper extensibility
  • automation and scriptability
  • integration with software dev workflows

The question is not whether one tool wins universally. The more responsible question is which workflow is stronger for which kind of problem.

Cost, access, and institutional adoption

Cost is part of the real comparison, especially for departments deciding whether to broaden access to e-learning production. As of April 17, 2026, Articulate's official pricing page listed Articulate 360 AI at roughly $1,449 USD / user / year for Personal and $1,749 USD / user / year for Teams, with academic discounts and other plan variations noted on the pricing page. Because pricing changes, verify current numbers directly before using them in institutional planning.

By contrast, an AI-assisted code-authored workflow usually shifts cost into a different structure: model or API usage where applicable, staff time, hosting or deployment, and training and maintenance overhead. This does not mean the code-authored route is always cheaper. In some settings, staff time may outweigh license savings. But the cost model is different, and often more flexible.

AI model and workflow risk

Cost is not the only structural difference. A code-authored workflow that depends on Codex also inherits the reliability profile of whichever AI provider sits behind it. Pricing changes, model deprecations, output-quality shifts between model versions, and API outages all turn into production risks for your content pipeline if you plan naively. Traditional authoring tools carry their own version-lock risks, but the failure modes are different.

Concrete mitigations worth building in from day one:

  • Document the prompts, not just the output. Store the master schema and project-level prompts in the repository alongside the code; they are the recipe for rebuilding if the current model version drifts or disappears.
  • Pin model and library versions in any automation. "Use Codex" is not a reproducible build instruction; "Use Codex with this prompt against this model family, produce plain HTML/CSS/JS with no runtime dependency on the provider" is.
  • Keep outputs provider-independent. The final learning object should not call the model at runtime for a standard SCORM or web build. If it does (e.g., live tutoring), treat the provider as a production dependency with SLAs, not an authoring convenience.
  • Have a manual-fallback plan for small maintenance edits — tone tweaks, copy fixes, rubric changes. If the only way to update a typo is a round-trip through the model, the content is fragile.

What this document still does not prove

It is equally important to say what this document does not yet establish. The tutorial does not yet provide:

  • a true side-by-side build of the same learning object in Articulate and in a code-authored workflow
  • measured comparative results across the three evaluation layers
  • a full institutional cost comparison based on actual staffing and deployment assumptions
  • empirical evidence showing cross-platform equivalence across multiple LMS environments

This document should be read as a rigorous workflow and evaluation framework, not as a completed proof that code-authored learning objects have already replaced Articulate across the board.

Project prompt

Build the same learning object twice — once in Articulate and once in an AI-assisted code workflow — then compare both versions across three layers: interaction quality, packaging and LMS runtime behavior, and maintainability over time. Keep the learning goal constant; vary only the authoring workflow. This turns the current document into a scaffold for empirical work rather than a claim that has already been settled.

The non-negotiable prompt components

The design contract between pedagogy and implementation.

Before your first major prompt, define these components explicitly. They are not optional if you want reliable results. If you omit them, Codex will fill the gap by inference — and that is often where instructional mismatch begins.

Just as important: the same component names used in your planning notes should also appear in your actual prompts. The conceptual explanation and the prompt structure should match one another exactly. Do not switch halfway into vague language like "make it nice" or "build something interactive" — that breaks alignment between the instructional concept and the implementation request.

Most weak AI-assisted build processes do not fail because the model "cannot code." They fail because the build request does not preserve the educational logic with enough precision. If the learning goal is described vaguely, the resulting interaction may be visually appealing but instructionally shallow. If tracking is not stated, the activity may appear functional until you try to package it for SCORM or restore state after re-entry. These components are not administrative metadata — they are the design contract between intent and implementation.

5.1 Learning goal

What should learners understand, practice, or decide differently after using the content?

  • Understand the greenhouse effect
  • Compare country-level educational performance patterns
  • Practice arithmetic fluency
  • Make better triage decisions in a simulated case
  • Identify misleading data displays

The learning goal should be written as a learner outcome, not as a production description. "Build a cool simulation" is not a learning goal. "Help learners understand how variable changes alter system behavior" is.

Write the goal so that transfer is visible: name the context in which the learner should later use what they practice. "Compare country-level educational performance patterns to inform a policy brief" is stronger than the same sentence without the final clause, because it tells Codex (and you) which interactions are load-bearing and which are decoration. If you cannot describe where the skill will be reused, the activity is at risk of being inert.

5.2 Audience

Who is the learner? Examples: middle school students, undergraduates, graduate students, K–12 teachers, healthcare trainees, public-facing adult learners. The same interaction can feel appropriate or completely wrong depending on audience. If you omit it, Codex tends to default toward a generic professional-user tone, which may not fit classroom, outreach, or novice-training contexts.

5.3 Content type

Explainer, quiz, scenario, simulation, data visualization, game. Content type is not just a label — it implies a different instructional rhythm. An explainer favors sequence. A quiz favors item progression and feedback timing. A branching scenario favors consequence and path memory. A simulation favors parameter response and interpretability.

5.4 Core interactions

What does the learner actually do? This should describe learner actions, not interface features. Saying "include a left panel and a timeline" describes furniture. Saying "the learner compares states across time and selects one case for deeper inspection" describes the actual learning action.

5.5 Feedback or assessment behavior

How will the learning object respond? Many e-learning builds become instructionally thin because they stop at interactivity and never specify response behavior. A good learning object does not merely wait for input — it reacts in a way that supports learning. That reaction could be corrective, explanatory, reflective, comparative, cumulative, or state-based.

State whether feedback is formative (designed to shape the next attempt — hints, explanatory responses, retry, worked examples) or summative (designed to measure achievement — final score, one attempt, no mid-activity guidance), and prompt Codex accordingly. "Allow retry with explanatory feedback, do not record a graded score" is a formative spec. "Single attempt, submit-and-lock, record final score to the LMS" is a summative spec. The same surface-level quiz UI can serve either purpose — the behavior you ask for determines which.

5.6 Content or data source

Is Codex authoring the material, scaffolding around existing material, or using mock content temporarily? Those are very different situations. If you already have a dataset, a scenario script, or a set of quiz items, say so. If placeholder content is acceptable, say that too.

5.7 Tracking and persistence requirements

What should be remembered, measured, or reported? Score, current step, selected answers, selected scenario path, visited states, completion, time on task, selected filters, resume state. This is especially important for SCORM or any learning-engineering use case, and it is one of the most expensive components to retrofit later.

5.8 Deployment target

Moodle SCORM, standard website, or both. This is not just an operational detail — it affects how state is stored, how assets are organized, how dependencies are handled, and how testing should be done.

5.9 Technical constraints

Examples: single HTML file, plain HTML/CSS/JS only, no external libraries, specific library version, SCORM 1.2 compatibility, mobile support, local-only assets. These are not there to sound rigorous — they are there to prevent later mismatches.

If learners will reach the object on phones — assume they will — say so explicitly: "Must work on a 360×640 viewport, touch targets at least 44×44 CSS pixels, no hover-only interactions, no layout below 320 CSS pixels of width." Mobile support specified as an afterthought tends to produce content that technically renders but is unusable on the device the learner actually holds.

5.10 First-version scope

What should the first build optimize for? Working structure over polish. Placeholder content is acceptable. Mock data is acceptable. Keep code easy to revise. Do not package for SCORM yet. This is one of the most effective ways to preserve momentum — it tells Codex what not to optimize too early.

The best overall workflow

Eight stages from definition to preserved stable versions.

Here is the most reliable end-to-end sequence.

Stage1

Define the learning artifact

Clarify goal, audience, content type, core interactions, feedback, content source, tracking, deployment target, technical constraints, and first-version scope.

Stage2

Ask for a functional first version

Prioritize structure, behavior, and correctness of the core interaction. Do not prioritize final polish, advanced packaging, or complex tracking.

Stage3

Open and review the output

Use the browser. Look at layout, readability, flow, clarity of instructions, feedback behavior, performance. Interactive content should not be reviewed only by reading code — you need to see it, click through it, and notice where friction appears.

Stage4

Give concrete revision feedback

Focus on categories: learning clarity, layout, tone, pacing, tracking, technical bugs.

Stage5

Stabilize the core learning experience

Deliberately resist the temptation to package too soon. If the learning experience is still unstable, SCORM packaging will only hide the real problem under runtime complexity.

Stage6

Add runtime packaging or persistence

Only after the main experience works: add SCORM packaging, resume logic, score/completion logic, and localStorage for web persistence.

Stage7

Test in the real target environment

For SCORM: upload to Moodle, launch, interact, exit, re-enter, verify state restoration. For web: open from local preview or hosting, verify persistence and layout.

Stage8

Split stable versions

Preserve the SCORM version, the web version, and (optionally) a source or template version. Do not keep rewriting one file forever for every use case.

10

How to review intermediate builds

Four review lenses: learning, interaction, design, technical.

After Codex creates a version, stop and review it before requesting more features. Review is how you decide whether the object is becoming more teachable and more usable for the learners you actually care about.

10.1 Learning quality

  • Is the purpose clear?
  • Does the activity actually support the stated learning goal?
  • Are instructions understandable? Feedback useful? Pacing appropriate?
  • Does the interaction help learning rather than distract from it?
  • Does the activity build toward transfer — could the learner apply what they practiced here in a new context, not just on this screen?
  • Is feedback behavior consistent with the intended assessment role — formative (hints, retry, explanations) or summative (single attempt, recorded score), not an accidental mix?

This category should come first on purpose. A build can be technically correct and aesthetically polished while still missing the real learning need.

10.2 Interaction quality

  • Is the flow intuitive? Are the controls clear? Is the next step obvious?
  • Are errors recoverable?
  • Does the learner understand what happened after each action?

Confusion at the control level often gets misdiagnosed as a content problem.

10.3 Design quality

  • Is the layout balanced? Content readable? Tone appropriate?
  • Does the design support attention rather than scatter it?

Educational interfaces often benefit from restraint more than spectacle.

10.4 Technical quality

  • Does the app load correctly? Is state preserved?
  • Visible runtime errors? Consistent logic? Works in the target container?

10.5 Factual and AI-content validation

Codex will confidently produce plausible-but-wrong facts, fabricated citations, misleading data, and simulation rules that look reasonable but do not match the actual system being modeled. None of the four review lenses above catch those by default. Add a factuality pass for any build that teaches content, not just skills.

  • Source-check every factual claim rendered in the UI — definitions, statistics, historical dates, scientific mechanisms, quoted regulations. If the prompt did not supply a source, assume the output is unsourced and verify externally.
  • Inspect simulation and scenario rules. For branching scenarios, check that consequences are pedagogically honest rather than convenient. For simulations, verify that the underlying model (formulas, probabilities, parameter ranges) matches the domain, not Codex's guess.
  • Re-read generated feedback and explanations. Feedback that says "good job, the greenhouse effect traps infrared radiation" is only useful if the explanation is correct — a subtle error inside supportive tone is harder to detect than a visibly broken control.
  • Cross-check citations and references. If Codex outputs author names, years, or titles, confirm they exist before they reach the learner. Fabricated references in learning materials are a disclosure-worthy error.
  • Keep a short errata log alongside the source files. When you catch an AI-introduced mistake, record what it was, how you found it, and what you changed — this is both a quality trail and a training signal for how to prompt better next time.
Rule

No factual claim reaches learners without a named human reviewer. "Codex said so" is never a source.

Most useful follow-up prompts

Seven recurring revision categories.

12.1 Structure

Please revise the structure of the activity. Keep the topic and content, but change the interaction flow so the learner moves through: (1) introduction, (2) guided practice, (3) independent attempt, (4) summary.

12.2 Feedback

Improve the feedback logic. I want the learner to receive immediate explanatory feedback after each action, not only a correctness label. Keep the scoring model, but make the feedback more instructionally useful.

12.3 Layout

Redesign the layout so the learning task is more central. Reduce visual clutter, improve spacing, and make the main interaction area dominant. Keep the same content and logic.

12.4 Tone

The current tone is too dark and cinematic. Please make the interface brighter, clearer, and more appropriate for educational use. Reduce glow, visual noise, and unnecessary effects.

12.5 Tracking

Add interaction tracking for: total clicks, completed steps, selected responses, time spent in each section. Store the tracking so it can later be mapped to SCORM or local persistence.

12.6 Accessibility

Improve accessibility. Please review color contrast, keyboard reachability, focus visibility, button labeling, and screen-readable structure. Do not change the instructional logic unless needed.

12.7 Localization

Convert the UI and learner-facing text to English. Keep the data and interaction logic unchanged. Update labels, buttons, instructions, and feedback text consistently.

How to ask for SCORM packaging

Translation step from interactive content to LMS-compatible courseware.

Once the learning object works well, ask Codex to package it for SCORM. Packaging is not merely a zip operation — it forces you to decide which parts of learner activity count as progress, which parts should be restorable, and how the runtime should behave on exit and return.

Default baseline

Unless otherwise noted, this tutorial assumes SCORM 1.2 as the default. It remains the safest starting point because it is broadly supported and easier to reason about when you are first learning how packaging, status, score, and resume state fit together. SCORM 2004 is real and still relevant, but should not be casually mixed into an introductory workflow.

SCORM is a wrapper, not a rendering style

SCORM does not tell you whether your learning object should be plain HTML, a quiz UI, a simulation, a D3 chart, a Three.js scene, or a framework-based interface. It is primarily a packaging and runtime convention for LMS delivery.

The rendering approach should be chosen based on what best supports the learning task. SCORM comes afterward as the delivery wrapper.

LTI integration (Canvas-first path)

When SCORM is the wrong wrapper — what to build instead.

If your primary LMS is Canvas, Brightspace (D2L), or an institution that has standardized on modern launch protocols, SCORM is often not the best wrapper. The Learning Tools Interoperability (LTI) family of specifications — especially LTI 1.3 plus LTI Advantage — is the path most higher-education LMSs actively favor for non-trivial interactive content. This section explains when LTI is the right default, what it gives you that SCORM does not, and how to ask Codex to build an LTI-launched tool.

What LTI is (and is not)

LTI is a launch-and-messaging specification, not a packaging format. Where SCORM wraps learning content into a zip that the LMS serves locally, LTI lets an LMS launch an externally hosted tool, pass signed identity and context claims, and (with LTI Advantage) exchange grades, roster information, and deep-linked resource choices. The tool runs on your own infrastructure; the LMS becomes the launcher, not the host.

When LTI is the better default

Choose LTI over SCORM when any of the following is true:

  • the primary LMS is Canvas, Brightspace, or Blackboard Ultra, and the content is more than a simple quiz or explainer
  • you need richer event data than SCORM's CMI model can express — branching analytics, per-question timing, open-text responses
  • the content will be updated often (server-side deploys beat re-uploading zips to every course shell)
  • you need roster awareness — group assignments, cohort-level behavior, instructor-vs-student views
  • you want the instructor to choose which learning object to embed from a library at placement time (Deep Linking)
  • you need single sign-on with identity you can actually verify

Stay with SCORM when: you need portability across LMSs without running a server, your institution will not whitelist your hostname, the content is a one-shot module with simple completion/score, or offline playback matters.

Practical rule for Canvas-first programs

If your program's primary LMS is Canvas and the learning object does anything richer than a short quiz, default to LTI 1.3. SCORM remains a valid fallback for portability, but it should be the compromise choice, not the reflex one.

Minimum viable LTI 1.3 tool

An LTI 1.3 tool is, at its smallest, a web application that:

  1. exposes a public JWKS endpoint advertising your tool's public keys
  2. accepts an OIDC login initiation POST from the LMS
  3. validates the signed launch JWT (audience, issuer, nonce, signature against the LMS's JWKS)
  4. renders the learning experience, scoped to the launch's user and context
  5. optionally calls back to AGS to post a score, or NRPS to fetch the roster

Well-maintained server libraries exist in most ecosystems — for example, ltijs (Node.js), PyLTI1p3 (Python), and the IMSGlobal/lti-1-3-php-library (PHP). Prefer a library over hand-rolling JWT validation; the security surface is non-trivial.

Prompting Codex for an LTI-launched tool

Build a minimal LTI 1.3 tool using [library of choice]. The tool should: expose /jwks, /login, and /launch endpoints; validate the incoming OIDC-launched JWT against the platform's JWKS; render the learning interaction scoped to the launched user and context_id; and post a score back via Assignment and Grade Services when the learner completes the activity. Include a README that lists exactly which platform registration fields (client_id, deployment_id, auth URL, token URL, JWKS URL) must be configured on the LMS side. Do not embed LMS keys inline — read them from environment variables.

Testing LTI locally

Two tools make LTI development tractable without waiting for an institutional Canvas instance:

  • IMS Global LTI Reference Implementation (https://lti-ri.imsglobal.org) — a free platform-side test harness for LTI 1.3 launches, Deep Linking, AGS, and NRPS.
  • Canvas free-for-teacher or an institutional Canvas test environment — both let you register a Developer Key and install your tool via a full LTI 1.3 flow.

A realistic test pass includes: OIDC login, JWT validation with correct and intentionally-bad signatures, score round-trip via AGS, roster fetch via NRPS, Deep Linking selection, and launch from a non-instructor role to verify role-scoped behavior.

Security reminder

LTI 1.3 is OAuth 2.0–based and carries real identity. Treat the private key that signs your tool's assertions like any production secret: rotate it, keep it out of source control, and log nothing that includes the launch JWT in full.

Read this section as "how to do it when the LMS asks for a launchable tool." Most Canvas-first programs will spend more time in this section than in the SCORM chain; most Moodle-first programs will spend more time in the SCORM chain. Accessibility (WCAG 2.1 AA)

A legal floor for public institutions, a professional baseline for instructional designers.

For US public universities, WCAG 2.1 Level AA conformance is not a stylistic preference — it is a legal obligation under Section 504 and the ADA, and it is increasingly a procurement requirement for courseware. For instructional designers, it is also a core professional competency. Treat this section as a pre-release checklist, not a "nice to have."

The default target in this tutorial is WCAG 2.1 AA. WCAG 2.2 became a W3C Recommendation in October 2023 and carries all 2.1 criteria forward plus nine additional AA items; projects with room for it should target 2.2 AA as a forward-looking baseline. Either way, do not ask Codex to "make it accessible" and stop there. Ask against specific criteria, and verify by testing.

Core accessibility prompt

Make this learning object conform to WCAG 2.1 Level AA. Specifically: every interactive control must be keyboard-operable with a visible focus indicator; text contrast must be at least 4.5:1 (3:1 for large text) and UI/graphical contrast at least 3:1; every non-text element needs a text alternative; every video needs captions and, where feasible, an audio description or transcript; every form control and custom widget needs an accessible name and role via native semantics or ARIA. Do not change the instructional logic. When a tradeoff is required, explain it rather than silently degrading the interaction.

  1. Keyboard pass: unplug the mouse and complete the full learner task. Note every control you cannot reach, activate, or exit.
  2. Screen-reader pass: run through the activity with NVDA (Windows, free), VoiceOver (macOS), or Orca (Linux). Confirm that every control announces a name, role, and current state.
  3. Contrast pass: check key foreground/background pairs with a tool such as the WebAIM contrast checker or the Chrome/Firefox devtools contrast panel.
  4. Automated pass: run axe DevTools or Lighthouse on the rendered page. Automated tools catch roughly 30–40% of issues — useful as a floor, not a ceiling.
Institutional reminder

If you are building for a US public university, document the conformance level you targeted, the criteria you tested, and any known exceptions. "We intended to be accessible" is not a defensible record. A short accessibility statement per learning object is far stronger.

Inclusive design beyond WCAG

WCAG 2.1 AA is a legal floor, not a ceiling. An object can be conformant and still be exclusionary. A few dimensions that do not appear on the WCAG checklist but materially affect whether the object works for real US learner populations:

  • Plain-language pass. After the instructional logic stabilizes, run the text through a readability check (Hemingway, Microsoft Editor, or the Flesch–Kincaid readout in your authoring tool). For undergraduate general-education courses, grade 9–11 is a defensible default; professional and graduate courses can go higher but should still avoid passive pile-ups and nested subordinate clauses. Non-native English speakers make up a large share of US higher-ed and corporate cohorts — plain language is not dumbing-down, it is load-reduction.
  • Cognitive load and time. Timed interactions, autoplaying animations, and mandatory read-then-answer sequences disadvantage learners with ADHD, dyslexia, anxiety disorders, or executive-function differences covered under Section 504. Provide pause, extend, or remove-timer options where a timer is not the skill being assessed.
  • Representation in examples. Names, scenarios, and images default toward a narrow demographic unless deliberately varied. For case-based content especially, ask Codex to propose a demographically diverse scenario set and then edit; the default distribution skews.
  • Low-bandwidth and shared-device contexts. Many working learners reach content on a phone over a cellular connection or on a shared family device. Heavy hero videos and autoplaying audio are a poor fit; offer a text-first path.
  • Neurodivergent-friendly defaults. Prefer explicit instruction states over ambient cues; give every interactive element a clear name and state; avoid motion-as-meaning without a static equivalent.

A short inclusive-design review — one reviewer, thirty minutes per object — tends to catch more shipping defects than another automated-tool pass once WCAG AA is already met.

Mobile and touch

Mobile support is part of the conformance baseline, not a separate concern. WCAG 2.1's 1.4.10 (Reflow) and 2.5.5 (Target Size, AAA) already imply it, and WCAG 2.2 promotes target size to AA (2.5.8). More importantly, in many programs the majority of learner sessions happen on phones.

  • Viewport meta tag present — <meta name="viewport" content="width=device-width, initial-scale=1">. Without it, mobile rendering degrades to a zoomed-out desktop view.
  • Touch targets ≥ 44×44 CSS pixels for every interactive control (WCAG 2.2 AA, 2.5.8). Small icon buttons and close-together radio buttons are the usual failures.
  • No hover-only interactions. Any information revealed on hover must also be reachable via focus or tap.
  • Reflow at 320 CSS pixels of width — vertical scrolling only, no two-dimensional scrolling. Test by narrowing the devtools emulator to a 320-pixel viewport (1.4.10 AA).
  • Be aware of the SCORM-in-iframe mobile reality. Most LMS mobile apps embed SCORM content in a constrained iframe with limited gesture support; complex drag-and-drop and multi-touch interactions often degrade there. Test in the actual LMS's mobile app, not just a phone-emulated desktop browser.

Privacy and data-leak awareness

SCORM packages are still web apps and carry ordinary web risks. If your project becomes more production-oriented, review the package for: unexpected external requests, hidden analytics calls, hardcoded secrets, learner identifiers embedded in state, and unsafe use of query strings or stored data.

Versioning strategy

Do not destroy the working version just to make the next experiment.

At minimum, keep one working source version, one SCORM version, and one web version. If the content grows, also keep an earlier stable checkpoint, a simplified demo, and a data-only reference file.

Preserve stable versions as soon as they work and treat later modifications as forks rather than overwrites.
Prompts are part of the source

Version the prompts that produced each stable build, not only the generated files. If the model later changes behavior, or the API becomes unavailable, the documented prompts are your rebuild recipe — without them, the most expensive part of the workflow (the accumulated design decisions) is stranded inside throwaway chat history.

Credentialing and portability

If the learning object is part of a stackable credential, a micro-credential program, or a corporate skills-ladder, version control extends past the source tree into the credentialing layer. Three specifications are worth knowing by name:

  • Open Badges 2.x (1EdTech / IMS Global). The dominant US interoperability standard for portable digital badges. A badge is a JSON-LD document carrying the issuer, recipient, criteria, evidence URL, and (optionally) alignment to a published framework. Credly and Accredible are the most common US issuing platforms.
  • IMS Comprehensive Learner Record (CLR) 2.0. A structured format for multi-credential transcripts — higher-ed registrars use it to issue verifiable records that include courses, competencies, and co-curricular achievements. Parchment and the National Student Clearinghouse are common US CLR endpoints.
  • Open Badges 3.0 / W3C Verifiable Credentials. The emerging layer. Institutions piloting wallet-based learner credentials are migrating here. If your program has a multi-year horizon, design identifiers and criteria strings that will survive the 2.x-to-3.0 shift.

Three practical habits for the authoring workflow:

  1. Give every learning object a stable identifier (URI or UUID) from the first stable build, and never reuse it for a later revision. Re-issuing a "v1" badge against a silently updated object breaks the audit trail.
  2. Record the alignment — which published competency, accreditation standard, or job-role framework the object addresses — alongside the source. For US higher-ed, common frameworks are AAC&U VALUE rubrics, discipline-specific accreditor competencies, and state workforce boards. For corporate L&D, common frameworks are the employer's internal role model, NICE (cybersecurity), and the Lumina Foundation's Degree Qualifications Profile.
  3. When the object changes substantively (more than copy edits), issue a new version identifier and, if a badge was already awarded against the prior version, make the policy call explicit: does the existing cohort get the new badge, re-earn it, or receive a "v1.x-equivalent" notation?
Corporate L&D note

If the learning object contributes to regulated compliance training (OSHA, HIPAA, SOX, FDA 21 CFR Part 11, DoD 8570), the audit retention period for completion records typically exceeds the object's own revision cadence. Keep a frozen copy of each version that ever recorded a completion, not only the current HEAD — auditors will ask to see what the learner actually saw.

Final advice

Preserve design logic across stages.

The strongest collaboration pattern is not "Write everything perfectly in one shot." It is:

Good collaboration is not mainly about wording clever prompts. It is about preserving design logic across stages. Minimal quiz example

A small, readable, resumable learning object.

A fully written minimal example — not production-ready, but showing what a small resumable learning object can look like. One question at a time, immediate feedback, final score, resume-friendly state, external scormdriver.js.

<!doctype html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <title>Minimal SCORM Quiz</title>
</head>
<body>
  <main>
    <section class="card" aria-live="polite">
      <h1>SCORM Quiz Starter</h1>
      <div id="app"></div>
    </section>
  </main>

  <script src="scormdriver.js"></script>
  <script>
    const questions = [
      { prompt: "Which planet is known as the Red Planet?",
        choices: ["Mars","Venus","Mercury","Jupiter"],
        correctIndex: 0,
        explanation: "Iron oxide gives Mars its reddish appearance." },
      /* ... more items ... */
    ];

    const passingScore = 70;
    const state = { currentIndex: 0, answers: [], score: 0, completed: false };
    const scorm = window.scorm12;

    function init() {
      scorm.init();
      scorm.setStatus("incomplete");
      restoreState();
      render();
    }

    function restoreState() {
      const raw = scorm.getSuspendData();
      if (!raw) return;
      try { Object.assign(state, JSON.parse(raw)); }
      catch (e) { console.warn("bad suspend_data", e); }
    }

    function persistState() {
      scorm.setSuspendData(JSON.stringify(state));
      scorm.setLocation(state.completed ? "summary" : String(state.currentIndex));
      scorm.commit();
    }

    function answerQuestion(choiceIndex) {
      const q = questions[state.currentIndex];
      const isCorrect = choiceIndex === q.correctIndex;
      state.answers.push({ questionIndex: state.currentIndex, choiceIndex, isCorrect });
      if (isCorrect) state.score += Math.round(100 / questions.length);
      persistState();
      render();
    }

    function finishQuiz() {
      state.completed = true;
      const finalScore = Math.min(state.score, 100);
      scorm.setScore(finalScore, 0, 100);
      scorm.setStatus(finalScore >= passingScore ? "passed" : "failed");
      scorm.setExit("suspend");
      persistState();
      render();
    }

    window.addEventListener("beforeunload", () => {
      if (!state.completed) { scorm.setExit("suspend"); persistState(); }
      scorm.finish();
    });

    init();
  </script>
</body>
</html>

The full file lives under examples/minimal-scorm-quiz/index.html. This reproduces the instructional core.

Minimal scormdriver.js

A starter reference, not a full compatibility layer.
(function () {
  function findApi(win) {
    var maxTries = 20, current = win, tries = 0;
    while (current && tries < maxTries) {
      if (current.API) return current.API;
      tries += 1;
      if (current.parent && current.parent !== current) current = current.parent;
      else break;
    }
    current = win.opener; tries = 0;
    while (current && tries < maxTries) {
      if (current.API) return current.API;
      tries += 1;
      if (current.parent && current.parent !== current) current = current.parent;
      else break;
    }
    return null;
  }

  function createDriver() {
    var api = null, initialized = false;
    function safeCall(fn, fallback) {
      try { return fn(); }
      catch (e) { console.warn("SCORM call failed", e); return fallback; }
    }

    return {
      init: function () {
        if (initialized) return true;
        api = findApi(window);
        if (!api) { console.warn("No SCORM 1.2 API. Browser-only mode."); return false; }
        initialized = safeCall(function () { return api.LMSInitialize("") === "true"; }, false);
        return initialized;
      },
      finish:   function () { return api && safeCall(function(){return api.LMSFinish("")==="true";}, false); },
      get:      function (el) { return api ? safeCall(function(){return api.LMSGetValue(el)||"";}, "") : ""; },
      set:      function (el, v) { return api ? safeCall(function(){return api.LMSSetValue(el, String(v))==="true";}, false) : false; },
      commit:   function () { return api && safeCall(function(){return api.LMSCommit("")==="true";}, false); },
      setStatus:      function (s) { return this.set("cmi.core.lesson_status", s); },
      setScore:       function (r, mn, mx) { this.set("cmi.core.score.raw", r); this.set("cmi.core.score.min", mn); this.set("cmi.core.score.max", mx); },
      setLocation:    function (l) { return this.set("cmi.core.lesson_location", l); },
      setSuspendData: function (v) { return this.set("cmi.suspend_data", v); },
      getSuspendData: function ()  { return this.get("cmi.suspend_data"); },
      setExit:        function (v) { return this.set("cmi.core.exit", v); }
    };
  }

  window.scorm12 = createDriver();
})();
